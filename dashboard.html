🤖 Network Health Assistant

📖 Reading this dashboard — what does each metric mean?

Throughput — how much data your link moves per second

Download (↓ down_mbps) · unit Mbps (megabits/sec) · higher is better
How fast you can pull data from the internet — video, software updates, web pages. Reported by a speedtest cycle to a nearby server.
Rough bands: ≥25 = single 4K stream OK · ≥100 = comfortable for a few users · ≥500 = fiber territory. Depends on household size.
Upload (↑ up_mbps) · unit Mbps · higher is better
How fast you can push data to the internet. Usually much lower than download on cable/DSL. Matters for video calls, cloud backups, sending images/codebases to Claude.
Rough bands: ≥5 = HD video calls · ≥20 = comfortable for Claude/agent workloads · ≥50 = large file uploads feel snappy.

Latency — how long a packet takes to arrive

RTT (round-trip time) · unit ms (milliseconds) · lower is better
Time for a tiny test packet to travel from your computer to a server and back. Reported as min/avg/max per anchor target — see "Anchor targets" below.
Heavily distance-dependent. Same city: <10 ms. Same country: <30 ms. Same continent: <80 ms. Transatlantic: 120–180 ms. Asia from EU: 200–300 ms. Compare against the right anchor — see below.
Jitter · unit ms (standard deviation of RTT) · lower is better
How much the latency wobbles between consecutive packets. Steady = good; spiky = bad. Wifi typically has higher jitter than ethernet.
<5 ms rock-solid (competitive gaming, real-time audio) · <30 ms fine for video calls · >30 ms calls go choppy, games rubber-band.
Packet loss · unit % · lower is better
Percentage of test packets that don't arrive. Every modern application can handle a tiny amount, but loss compounds badly on long-lived connections.
0% ideal · <0.1% essentially perfect · >1% visible: streams stutter, Claude SSE streams drop mid-generation · >3% severe (bad cable, wifi interference, ISP problem).

Bufferbloat — how much the link clogs under load

Bufferbloat (increase_ms) · unit ms · lower is better
RTT measured while the speedtest is saturating the link, minus the idle RTT. Tells you how much extra latency your router adds when traffic is heavy — the single most important metric for gaming and video calls feeling responsive while someone else is downloading. Most consumer routers without Smart Queue Management (SQM) score C–F.
Waveform's well-established grading:
A+<5 msexcellent
A5–30 msgood
B30–60 msacceptable
C60–200 msgaming/calls suffer
D200–400 mspoor
F>400 mslink unusable under load

Connection setup & capacity

TLS handshake RTT (tls_rtt_avg_ms) · unit ms · lower is better
Time to set up a fresh secure HTTPS connection (the slow part before any real data flows). Typically 2–3× the raw RTT because TLS needs multiple round-trips. This is the cost an agent pays every time it opens a new connection.
From PL to Anthropic's edge: 60–100 ms is normal.
Concurrent TLS (concurrent_tls.ok) · unit count / 10 · higher is better
How many out of 10 simultaneous TLS connections succeeded. Tests whether your NAT/firewall caps parallel connections. Matters when running multiple Claude agents in parallel (each holds at least one long-lived connection).
10/10 = no ceiling. Anything less suggests something throttling fan-out.

Anchor targets — why we ping four different servers

  • anthropic — Anthropic's own anycast edge. The number that matters directly for Claude Code.
  • pl_local (wp.pl) — A Polish server. If this is slow but others are fine, you're slow even before international transit. If this is fast and others aren't, the problem is your ISP's peering.
  • eu_hub (Frankfurt) — Where ~all Polish international traffic passes through. Bad peering here = bad everything outside Poland.
  • us_east (Virginia) — Raw transatlantic transit baseline. Anthropic mostly serves you from EU edge, so this is more of a reference.

Workload verdicts — pass/fail per use case

Each verdict combines several metrics into a single OK / FAIL:

claude — upload ≥20 Mbps · packet loss <1% · RTT to Anthropic <200 ms
streaming_hd — download ≥5 Mbps
streaming_4k — download ≥25 Mbps · packet loss <1%
gaming — EU-hub RTT <80 ms · EU-hub jitter <5 ms · bufferbloat <30 ms
video_calls — down/up ≥5 Mbps · jitter <30 ms · loss <1%

OK = all conditions met · FAIL <reason> = one or more violated · UNKNOWN = data missing.

Context-dependent thresholds — flagging where these defaults could be wrong for you:
  • Throughput "good" bands assume a 1–2 person household. Larger household streaming 4K + gaming + cloud backups wants more.
  • The claude upload threshold (≥20 Mbps) assumes you'll run multiple parallel agents. A single agent works fine at ≥5 Mbps.
  • Gaming jitter <5 ms is tuned for competitive FPS. Casual gaming is OK at 10–15 ms.
  • Loss thresholds in the top "Latest status" panel use [0.5%, 1%] — 0% is the real healthy state. Any sustained loss above 0% is worth investigating, even if the panel still shows green.

About the smoke-test rows in your data

Not a feature. The smoke-test label in your logs is leftover from initial development (running ./bench.sh smoke-test to verify the script worked before real monitoring started). You can safely:

  • Filter it out via the label dropdown in the header, or
  • Delete it: rm logs/smoke-test.*

You can however use a single bench run as a real smoke test any time — e.g. on a fresh machine, after editing the script, or to sanity-check a new connection before starting an overnight monitor:

./bench.sh smoke-test
cat logs/smoke-test.jsonl | python3 -m json.tool

Pass = the JSON line has all five workload verdicts populated, throughput.down_mbps and up_mbps are numbers (not null), and bufferbloat.grade is a letter. That means every probe ran end-to-end and parsed correctly.
Fail = any of those fields is null. The script printed phase headers [1/5]…[5/5] to stderr — whichever phase didn't produce data is your culprit. Common causes: speedtest not installed, ping target unreachable, network down.
